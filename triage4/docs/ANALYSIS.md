# Анализ черновиков и оценка проекта

Документы, с которыми велась работа:

- `Branch · Branch · Branch · Обзор проекта svend4_meta2.md` (≈12 500 строк)
- `Branch · Branch · Branch · Обзор проекта svend4_meta2.txt` (тот же текст)
- `Branch · Branch · Branch · Обзор проекта svend4_meta2.json` (JSON-представление диалога)

Все три файла содержат **одно и то же содержание** в разных форматах: это лог
переписки `you asked / chatgpt response`, где итеративно развивается идея
четвёртого репозитория `triage4`, объединяющего `meta2 + infom + in4n` в
stand-off triage stack под DARPA Triage Challenge Event 3.

## Что в черновиках фактически есть

1. Сравнительный разбор репозиториев `meta2`, `infom`, `in4n` (роль каждого в
   экосистеме).
2. Разбор задачи DARPA Triage Challenge: Gate 1–4, human-machine teaming.
3. Архитектурный blueprint проекта `triage4` — слои, папки, классы.
4. Полный «minimal bootstrap» репозитория `triage4` в виде Python + TypeScript
   сниппетов (это самая ценная инженерная часть).
5. Вторая итерация bootstrap: добавлены `body_regions`, сигнатуры дыхания и
   кровотечения, страницы MapPage и ReplayPage.
6. «Фрактальная» K3-матрица 3×3: подуровни 1.1–3.3 с распределением ролей
   `meta2 / infom / in4n` по каждой ячейке.

## Плюсы черновиков

- **Архитектурная связность.** Идея `meta2 → infom → in4n → triage4`
  сформулирована последовательно: каждый слой имеет роль, названия классов и
  логику переиспользования.
- **Реалистичное MVP-мышление.** Автор не пытается сразу написать полный
  медицинский стек — первый спринт сознательно ограничен simulation-first
  подходом: FastAPI + небольшой React-дашборд + synthetic_benchmark.
- **Прямой мэппинг на DARPA gates.** Каждый модуль привязан к Gate 1–4 и HMT
  lane — это делает проект грантопригодным.
- **Фрактальная K3-матрица** — сильный архитектурный ход. Три уровня и три
  «режима» (`.1 / .2 / .3`) делают архитектуру самоподобной и расширяемой.
- **Код в черновиках синтаксически почти готов.** Python-сниппеты
  используют современные возможности (dataclasses, `X | None`, `from __future__
  import annotations`). FastAPI-эндпоинты валидны. React-код на чистом Vite.
- **Декларативный disclaimer.** В README явно написано «не медицинское
  устройство» — это важно для dual-use темы.

## Минусы черновиков

- **Формат.** Три файла по ~400 Кб с одним и тем же содержимым в md/txt/json —
  это не исходники, а лог переписки. Такая структура не запускается и не
  переиспользуется.
- **Дублирование.** Архитектура описана трижды (сначала обзор, потом
  `ARCHITECTURE.md`-черновик, потом K3-матрица). Нет единого источника правды.
- **Смешение концепций.** В рамках одного файла соседствуют: обзор `meta2`,
  сравнительные таблицы, архитектурный blueprint, bootstrap-код, K3-матрица,
  обсуждения DARPA, HCI и гранта. Нужно расщепить на отдельные документы.
- **Устаревший `on_event("startup")`.** В FastAPI 0.115+ это deprecated —
  предпочтителен `lifespan`. Для MVP-каркаса допустимо, но стоит отметить.
- **Стабовые имплементации.** `PersonDetector` / `PoseEstimator` заявлены, но
  в черновиках не реализованы. Это нормально для bootstrap, однако требует
  явного плана замены.
- **Нет Phase 2–7 на уровне кода.** Roadmap расписан на 7 фаз, но реально в
  черновиках присутствуют только Phase 1–2 и первая половина Phase 3.
- **ROS2 / MAVLink / Spot-мосты упомянуты, но отсутствуют.** Нужно либо
  оставить как явные placeholder-интерфейсы, либо вынести в отдельную фазу.
- **Нет CI / линтера / mypy.** Для research-проекта терпимо, но в
  медицинско-автономной теме это уже слабое место.

## Что было сделано на основе черновиков

1. Один большой markdown разбит на **рабочий Python-пакет `triage4`** из
   множества маленьких файлов (~30 модулей, каждый < 100 строк).
2. Добавлены недостающие компоненты, обозначенные в черновиках, но не
   написанные: `perfusion_signature.py`, `fractal_motion.py`,
   `trauma_assessment.py`, `explainability.py`, `mission_graph.py`,
   `updates.py`, `task_allocator.py`.
3. Реализована первая «диагональ» K3-матрицы: `semantic/`, `state_graph/`,
   `triage_temporal/`, `tactical_scene/`, `mission_coordination/`,
   `world_replay/`.
4. Добавлены адаптеры `integrations/meta2_adapter.py`,
   `integrations/infom_adapter.py`, `integrations/in4n_adapter.py` с
   минимальной рабочей реализацией.
5. Добавлены **тесты**: rapid triage, сигнатуры, body regions, graph,
   semantic+state, mission+replay.
6. Web UI разбит на отдельные файлы: `types.ts`, `util/priority.ts`,
   `pages/MapPage.tsx`, `pages/ReplayPage.tsx`, `pages/CasualtyDetail.tsx`,
   `App.tsx` — вместо одного монолитного `App.tsx`.
7. Добавлены документы: `ANALYSIS.md`, `ARCHITECTURE.md`, `ROADMAP.md`,
   `darpa_mapping.md`.

## Оценка по шкале готовности

| Критерий                      | До (черновик)         | После (этот репозиторий) |
|-------------------------------|-----------------------|--------------------------|
| Запускаемость                 | нет (только текст)    | `pytest`, `uvicorn`, `npm run dev` |
| Модульность                   | один ~12k-строчный md | ~50 файлов по 20–120 строк |
| Покрытие тестами              | нет                   | 6 тестовых файлов, 20+ проверок |
| DARPA-mapping                 | в тексте              | выделен в `docs/darpa_mapping.md` |
| K3-матрица 3×3                | идея                  | есть как каталоги-слои |
| Адаптеры `meta2/infom/in4n`   | интерфейсы            | минимальные реализации |

## Рекомендации на следующие шаги

1. Заменить FastAPI `on_event("startup")` на `lifespan`-контекст.
2. Подключить реальный upstream `meta2` (fractal module) через
   `Meta2SignatureAdapter`.
3. Добавить CI: `pytest`, `ruff`, `mypy`.
4. Реализовать `sim/scene_generator.py` и `sim/sensor_degradation.py`.
5. Подтянуть diagonal 2 из K3-матрицы (`1.2 → 2.2 → 3.2`) в полном объёме.
6. Сделать replay dataset сохраняемым на диск и воспроизводимым.
